# /gemini-analyze

Uses Gemini CLI's 1M token context window to analyze the entire codebase at once — no chunking required.

## Usage

```
/gemini-analyze                        # full codebase analysis
/gemini-analyze src/payments           # scoped to a specific module
/gemini-analyze src/auth src/users     # scoped to multiple modules
```

## Commands

### Full codebase analysis

```bash
gemini -p "$(cat GEMINI.md)

Analyze this codebase with focus on:
1. Architectural consistency with the decisions documented above
2. Technical debt: duplicated logic, unclear responsibilities, high coupling
3. Naming and structural inconsistencies across modules
4. Missing abstractions or over-engineered areas

$(find src -type f | sort | xargs cat 2>/dev/null)"
```

### Scoped to a specific module

```bash
gemini -p "$(cat GEMINI.md)

Analyze the [module] module:
1. Internal consistency and single-responsibility adherence
2. Integration points — how it couples with other modules
3. Naming, structure, and convention compliance
4. Technical debt and improvement opportunities

$(find src/[module] -type f | sort | xargs cat)"
```

## Output Format

Present the Gemini output structured as:

```
[GEMINI ANALYSIS] — [scope analyzed] — [date]

## Architectural Findings
...

## Technical Debt
...

## Inconsistencies
...

## Recommendations
...
```

## Using the Output

- The output is informational — no code is generated by Gemini
- Use findings as context for implementation decisions in Claude Code
- Significant architectural findings should update `GEMINI.md` and sync the anchor chat via `/agents-sync`
- Critical debt items can be captured as tasks or ADRs in `docs/decisions/`
